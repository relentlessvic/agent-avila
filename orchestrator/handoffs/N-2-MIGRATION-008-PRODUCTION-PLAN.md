## 3. Migration runner behavior

- **Runner:** `scripts/run-migrations.js` — 154 lines, committed in D-5.4.
- **Invocation forms** (per `package.json:11` and the runner's own usage block at line 14): the runner can be invoked locally through the package migrate script or directly with an explicit database URL. The production N-3 form would use the approved Railway runner invocation. This runbook intentionally omits the literal production command because `HANDOFF-RULES.md` forbids migration-apply commands or production Railway runner invocations inside packet artifacts.
- **Behavior:** reads every `migrations/NNN_*.sql` file matching `/^\d+_.+\.sql$/` (lines 49–51), string-sorts, looks up `schema_migrations` (lines 87–92), and applies every absent version in version order (lines 100–142).
- **Per-migration target?** **No.** The runner has no version-filter flag. **It applies all un-applied migrations sequentially.** This is the same behavior that caused Migration 006 to be applied as a side effect during Phase B.2a.
- **Filesystem-based, not git-tracked.** The runner reads `migrations/*.sql` from the working tree filesystem (lines 49–51), not from `git ls-files`. **Therefore a pre-flight `git ls-files` check alone is insufficient** — the runner would still pick up an untracked or dirty 009+ file. Pre-flight (§4) explicitly guards against this.
- **Safety properties (positive):** transactional per migration (`BEGIN`/`COMMIT`/`ROLLBACK`); checksum-locked (exit 3 on tampered already-applied migration); idempotent (re-runs are no-ops once applied).
- **Execution-context requirement (added in N-2d; Railway execution-surface correction added in N-2n after the second N-3 attempt at HEAD `3138e7f` halted with the identical DNS failure as Attempt 1).** The production Postgres add-on is reachable from inside Railway's **private network** via the Railway-internal hostname; that hostname does not resolve from environments outside Railway's private network. **Critical Railway networking distinction (added in N-2n, confirmed by Codex design review against Railway's documented behavior):** Railway has two materially different operator-facing execution surfaces for the production target, and they are NOT interchangeable for internal-hostname `DATABASE_URL`:
>
>   1. **Local execution surfaces with Railway environment-variable injection** — these include the Railway CLI's "run" mode (which executes a command locally with Railway service variables injected into the local process) and the Railway CLI's "shell" mode (which opens a local shell with Railway service variables available and sets a `IN_RAILWAY_SHELL`-style marker). Per Railway's documented behavior, **these surfaces inject environment variables only; they do NOT create a DNS tunnel into Railway's private network.** A local Node process invoked via either surface still uses the local OS / Node DNS resolver, which has no path to `*.railway.internal`. **A `DATABASE_URL` containing the Railway-internal hostname will NOT resolve from these surfaces.** Both N-3 attempts at HEAD `9ae139d` and HEAD `3138e7f` halted with `getaddrinfo ENOTFOUND postgres.railway.internal` and are canonical examples of this failure class — the runner was running locally even though Railway env vars had been injected. **These surfaces are explicitly REJECTED as primary N-3 execution paths for internal-hostname `DATABASE_URL`.**
>
>   2. **Railway-attached deployed-runtime shells** — these include attaching a shell session into a deployed Railway service container via the Railway CLI's "ssh" mode (`railway ssh`-class surface) or any equivalent Railway runtime surface that places the executing process *inside the Railway project's private runtime network*. From these surfaces, the `*.railway.internal` private DNS resolves natively because the executing process is part of the Railway project's private runtime network. **The Railway-attached deployed-runtime shell is the approved primary N-3 execution path** for this runbook. The deployed service in this project is `agent-avila-dashboard` (per `package.json` `scripts.deploy`); the operator attaches an SSH-class session into the deployed `agent-avila-dashboard` container, runs every §4 (i)–(xi) preflight check from inside that container, and invokes the migration runner from that same SSH session (no context switch — see §4(vii) same-session rule).
>
>   **Why a deployed-runtime shell, not a one-off Railway job/service or a public/proxy DB path?** A one-off Railway job would resolve the internal hostname natively but would require a deployment or service-config change (currently HARD BLOCKED for the N-3 production-action track) and is therefore an alternative strategy requiring its own separate design review, Codex review, and fresh Victor approval — see §15 alternative strategies (added in N-2n). A public-internet / proxy DB path would resolve from any context but introduces a new public-exposure attack surface (security regression) and is also an alternative strategy. The deployed-runtime shell is the only path that satisfies (a) executes inside Railway's private network so internal DNS resolves, (b) requires no deployment / no code change / no env / no secret read, and (c) can be gated by the existing same-session §4(vii) rules from inside the container.
>
> This runbook does NOT include literal commands, hostnames, ports, or URLs for any of these paths; the operator chooses and invokes the approved deployed-runtime-shell execution context per their own Railway access procedures, with the §4(vii) connectivity preflight (now tightened in N-2n to require the deployed-runtime-shell surface, and further tightened in N-2o to require deployed-runtime-shell availability and integrity) confirming reachability from inside the container before the runner is invoked.
>
> **Deployed-runtime-shell availability HALT branches (added in N-2o, mandatory).** Before §4(vii) is performed and before any runner invocation, the operator must independently confirm that the approved deployed-runtime-shell surface is actually available, that the deployed `agent-avila-dashboard` service has a running container that can be attached to, that any required shell-access setup (e.g., SSH key registration on the operator's Railway account) is in place, and that the deployed container exposes a usable interactive shell binary at the path the SSH-class surface invokes. **HALT** if any of the following is true: (a) the Railway-attached deployed-runtime-shell surface (e.g., `railway ssh`-class surface) is unavailable from the operator's environment — for any reason, including but not limited to: the Railway CLI subcommand is not installed or not authorized, Railway plan / project / service entitlements do not permit SSH-class shell access for this service, an outage / regional incident prevents the shell surface from establishing a session, or the operator's network blocks the underlying transport; (b) the deployed `agent-avila-dashboard` service has no running container available to attach to — for example: the service is in a CRASHED / SLEEPING / SUSPENDED / DEPLOYING / RESTARTING state, the latest deploy failed or is rolling out, no replica is currently RUNNING, or the dashboard surface that lists running replicas is empty; (c) shell-access credential / key setup fails — for example: SSH-key registration on the operator's Railway account fails, the registered key is not accepted by the Railway shell endpoint, the operator's local key agent is unavailable, or the key challenge is rejected; (d) the deployed container does not expose a usable interactive shell binary at the path the SSH-class surface invokes — for example: the container image does not include `/bin/sh` / `/bin/bash` or whichever shell the surface expects, the shell binary is present but not executable, or the surface immediately exits because no PTY can be allocated. In every case (a)–(d), do NOT proceed to §4(vii); do NOT invoke the migration runner from any other surface as a workaround; do NOT attempt to use the local Railway-CLI run / shell surfaces as a fallback (those are already REJECTED for internal-hostname `DATABASE_URL`); HALT and surface the unavailable-surface observation to the operator. Any future N-3 attempt requires the deployed-runtime-shell surface to first be made available and a fresh §11 gate satisfied (fresh Codex N-3 preflight PASS + fresh Victor approval naming the new HEAD).
>
> **Deploy-method source-identity gating (added in N-2t, mandatory; complements N-2n / N-2o execution-surface gating, distinct concern).** N-2n / N-2o reject local-execution surfaces (`railway run`, `railway shell`) for internal-hostname `DATABASE_URL` reachability — that is the *execution-surface* concern, governing where the runner runs. N-2t addresses a separate concern — *source-identity verification* governing how §4(x)(a) GAP-D Case 2 is satisfied. The Railway deploy-metadata fallback for source-revision identity (per §4(x)(a) GAP-D Case 2) requires a non-secret Railway surface that **exposes a verifiable full 40-character commit SHA** for the currently-running deploy. **`railway up` deploys (the project's documented deploy command per `package.json` `scripts.deploy`) typically do NOT record a commit SHA on Railway's dashboard or build-log surfaces** — they record only deployment ID, timestamp, build provenance (e.g., Nixpacks builder version, build duration), and image digest. **Image digest is NOT a commit SHA** (it is a hash of the resulting image bytes, deterministic for a given build but not directly mapping to a single source revision without an independent operator-attested mapping that this runbook does not approve). **Deployment ID is NOT a commit SHA** (it is a Railway-internal identifier for the deploy instance, not a repository revision). **Timestamp is NOT a commit SHA.** Without a verifiable full 40-character commit SHA from a non-secret Railway deploy-metadata surface, §4(x)(a) GAP-D Case 2 cannot be satisfied; operator-attested local-SHA mapping (e.g., "I ran `railway up` from working-tree state at commit X") is explicitly NOT a §4(x)(a) source-identity method per the runbook's existing canonical-source / non-secret-only / verifiable-from-Railway-deploy-metadata-only discipline. **Therefore: a deploy that lacks a verifiable full 40-character commit SHA on a non-secret Railway deploy-metadata surface is REJECTED as a valid N-3 source-identity surface.** The approved N-3 deploy path is **GitHub-push-tracked deploys** (or any equivalent deploy mechanism that produces a Railway-recorded full 40-character commit SHA on a non-secret surface). **HALT** before any N-3 attempt if: (a) the currently-running deploy of `agent-avila-dashboard` was triggered by a method that does not record a commit SHA (e.g., `railway up`, manual `railway redeploy` from a non-Git source, `railway up`-style image upload); (b) the Railway dashboard / build log / deploy-metadata API for the currently-running deploy does not expose a verifiable full 40-character commit SHA; (c) only short SHA, deployment ID, image digest, timestamp, or operator-memory-attested mapping is available for source-identity verification. To unblock N-3 in any of these cases, the operator must trigger a fresh GitHub-push-tracked deploy of the approved HEAD (or any equivalent that produces a Railway-recorded commit SHA), wait for that deploy to reach RUNNING, and re-run the §4(x)(a) GAP-D verification against the new deploy. This rejection is consistent with N-2n / N-2o's rejection pattern for local-injection surfaces: both rejection classes preserve the runbook's discipline that N-3 verification surfaces must come from non-operator-attested, Railway-side, non-secret records. **A `railway up`-style deploy method may still be acceptable for non-N-3 deploys** (this rejection applies only to N-3 source-identity verification, not to ordinary deploy operations).
